# WebUI → Native App 轉換計畫

目的：將現有的 `WebUI`（作為 web app）轉換成可跨平台發佈的應用程式（桌面或行動），同時保留原有功能並加入必要的原生整合。

分階段計畫如下，每個 phase 含目標、交付物與驗收條件：

---

## Phase 0 — 探勘與需求收斂（Discovery）
- 目標：蒐集使用者與業務需求，確認欲支援的平台（例如 Windows/macOS/Linux/Android/iOS）與優先順序。
- 交付物：需求清單、目標平台矩陣、風險評估報告、成功衡量指標（KPIs）。
- 驗收條件：利害關係人確認需求文件並同意下一階段技術選型。

---

## Phase 1 — 技術選型與架構設計
- 目標：決定技術棧（例如 Electron、Tauri、React Native、Capacitor 等），並設計應用整體架構。
- 交付物：技術比對報告、架構圖（包含前端、背景服務/代理、IPC/API）、安全與權限清單、原型概念驗證（POC）。
- 驗收條件：完成 POC，架構圖與技術選型經技術審查通過。

---

## Phase 2 — 模組化與後端/服務層調整
- 目標：將目前 `WebUI` 的後端或服務邏輯抽象化，便於在本地以服務或背景程序運行；建立穩定的本地 IPC 或 API 層。
- 交付物：重構後的服務層（可作為獨立進程或內嵌背景服務）、API/IPC 文件（schema 與契約）、測試用例。
- 驗收條件：本地背景服務可在開發平台以 CLI 或系統服務模式正常啟動，前端能透過 IPC 呼叫並取得預期回應。

---

## Phase 3 — 前端移植為原生/混合 App
- 目標：將前端 app 包裝為桌面或行動應用，並整合原生能力（文件存取、系統通知、硬體存取等）。
- 交付物：可執行的 App（開發版）、原生整合清單與實作、平台差異處理指南、使用者介面微調與無障礙檢查。 
- 驗收條件：目標平台可啟動 App，核心功能與 Web 版一致，並支援必要的原生 API。

---

## Phase 4 — 封裝、簽署與發佈管線（CI/CD）
- 目標：建立可重複的打包與發佈流程（含簽章、安裝程式、App Store/Marketplace 發佈流程準備）。
- 交付物：自動化 CI/CD 腳本、打包與簽署設定、發佈說明文件、安裝器或發佈包範例。
- 驗收條件：在目標平台產出可安裝的安裝包，並成功走完至少一次發佈流程（測試內部渠道）。

---

## Phase 5 — 測試、品質保證與監控
- 目標：建立完整的 QA 流程、E2E 測試、自動化回歸測試，並部署錯誤追蹤與使用者分析。 
- 交付物：測試計畫、E2E 測試套件、錯誤追蹤設定（Sentry 等）、監控與使用分析儀表板。
- 驗收條件：關鍵路徑測試通過，錯誤率與性能達到設定的 KPIs，回報機制運作正常。

---

## Phase 6 — 釋出、支持與持續優化
- 目標：正式釋出產品，建立支援流程並持續收集回饋進行優化。
- 交付物：發佈公告、支援手冊、版本管理策略、回饋蒐集與迭代計畫。
- 驗收條件：使用者回饋率、錯誤回報、更新頻率等指標達到預期目標，並能按計畫持續交付更新。

---

- Phase 1：2–4 週
- Phase 2：4–8 週（視代碼庫複雜度）
- Phase 3：4–12 週（依平台數量與原生整合複雜度）
- Phase 4：1–4 週
- Phase 5：2–6 週

交付物格式建議
- Markdown 文件存放在 `plans/webui-to-app`。
- 針對每個 phase 建立子文件（例如 `phase-1-architecture.md`）與對應的技術票（Issue / Project）。

後續建議
- 召開一次短會（30–60 分鐘）與利害關係人同步本計畫並取得優先順序。
- 由技術團隊建立 POC 並在 Phase 1 內驗證最關鍵的技術風險（例如選用的包裝方案能否滿足性能與安全需求）。
