### 疑難排解（TROUBLESHOOTING.md）

**目的**：系統化的問題診斷與解決流程

**結構**：
1. 診斷工具（healthcheck、日誌）
2. 問題分類（安裝/啟動/連線/執行/效能）
3. 每個問題的診斷步驟
4. 多種解決方案
5. 進階支援途徑

**診斷流程**：
```
症狀描述
  ↓
診斷步驟（確認問題根源）
  ↓
解決方案 1、2、3...
  ↓
驗證修復
```

**最佳實踐**：
- 從症狀出發，不假設用戶知道原因
- 提供可執行的診斷指令
- 按優先順序列出解決方案
- 包含預期的輸出範例

**範例**：
```markdown
### 問題：埠號被佔用

**症狀**：
```
Error: listen EADDRINUSE: address already in use :::5000
```

**診斷**：
```bash
lsof -i :5000
```

**解決方案 1**：停止佔用程式
**解決方案 2**：使用不同埠號
```

- 連結到相關文件
- 在文件底部提供導航

**範例**：
```markdown
---

**回到索引**：[用戶指南索引](USER_GUIDE_INDEX.md)

**相關文件**：
- [FAQ](FAQ.md)
- [疑難排解](TROUBLESHOOTING.md)
```

### 連結驗證

**自動化檢查**：
```bash
# 檢查損壞的連結
for file in *.md; do
  grep -o '\[.*\]([^)]*\.md)' "$file" | \
  grep -o '([^)]*.md)' | tr -d '()' | \
  while read link; do
    [ ! -f "$link" ] && echo "BROKEN: $link in $file"
  done
done
```

## 🎨 格式與風格

### Markdown 格式

**標題層級**：
```markdown
# H1：文件標題（每個文件一個）
## H2：主要章節
### H3：子章節
#### H4：細節（少用）
```

**程式碼區塊**：
```markdown
# 使用語言標記
```bash
echo "Hello"
```

```json
{"key": "value"}
```

**強調**：
- **粗體**：重要概念、UI 元素
- *斜體*：術語、引用

**使用表格**：
- 比較不同選項

## 🎓 經驗教訓

### 1. 從使用者視角出發

**錯誤做法**：
- 按照程式碼結構組織文件
- 使用技術術語沒有解釋

**正確做法**：
- 按照使用場景組織
- 用使用者的語言描述

### 2. 提供多層次文件

**新手需要**：快速開始
**進階用戶需要**：完整參考
**遇到問題需要**：疑難排解

不同用戶群需要不同深度的文件。

### 3. 實例勝於千言

一個可執行的範例比一頁抽象描述更有用。

### 4. 保持文件同步

程式碼變更時必須同步更新文件，否則會誤導使用者。

### 5. 建立回饋循環

文件永遠不會完美，需要持續根據使用者回饋改進。
